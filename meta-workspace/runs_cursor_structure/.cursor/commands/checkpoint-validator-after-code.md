# Приёмка кода (checkpoint-validator)

Запуск полного процесса разработки/приёмки после правок кода: **quick check → отдельный custom subagent → правки до `PASS`**. Используй `.cursor/agents/checkpoint-validator.md` как единственный регламент проверяющего контура.

## Цель

Выполнить контур приёмки для **текущего** этапа: отдельный read-only субагент со свежим собственным контекстом, проверка по источникам истины, цикл до **`PASS` без замечаний**, требующих действий.

## Источники истины (не расширять трактовку)

1. `task-brief.md`
2. `project-concept.md`
3. Текущий этапный запрос пользователя (не подменять планом из Plan mode)
4. Фактически изменённые файлы или дифф
5. Вывод последней **быстрой проверки** проекта

## Шаги

1. Запусти **быструю проверку** (линтер / тесты / сборка — что уместно для этого репозитория и объёма изменений).
2. Запусти **именно отдельный custom subagent** `checkpoint-validator` из `.cursor/agents/checkpoint-validator.md` со свежим контекстом: этапный запрос, выдержки из `task-brief.md` / `project-concept.md` при необходимости, список или дифф файлов, вывод quick check.
3. Не подменяй этот запуск самооценкой в текущем чате. Если отдельный вызов custom subagent недоступен, остановись и явно сообщи о блокере.
4. При `FAIL`, `WARN` или блокирующих пунктах в «что нарушено» / «риск для основного сценария» — внеси **минимальные** исправления, затем снова шаги 1–3.
5. Повторяй цикл, пока не будет **`PASS`** и не останется замечаний, требующих действий.

## Ожидаемый вывод в чат

После ответа отдельного custom subagent выведи результат **в формате полей**, заданном в `.cursor/agents/checkpoint-validator.md` (`вердикт`, `что подтверждено`, `что нарушено`, `риск для основного сценария`, `минимальное исправление`, `можно ли переходить дальше`). Если итог не `PASS` без замечаний — продолжай цикл исправлений и не закрывай этап.
