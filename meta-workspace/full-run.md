# Единый сценарий прогонов и live-demo

## Как использовать `full-run.md`

- `full-run.md` - единый и единственный редактируемый источник промптов и устного сопровождения для репетиций и финального live-demo.
- Этот файл дописывается и улучшается между прогонами; отдельные файлы с пакетами промптов не создаются без явного запроса.
- Любая правка существующего промпта или добавление нового промпта делаются только в соответствующем разделе этого файла.
- Каждый `run` моделирует одну полную презентацию в одном отдельном автономном workspace.
- Промпты для внутреннего run формулируются так, как будто перед агентом только текущий автономный проект.
- Для проверки этапа `N` нужно создать новый чистый run, последовательно применить промпты от `Этапа 1` до `Этапа N` включительно и затем соотнести состояние проекта с критериями этого этапа.
- Если внутри этапа есть несколько подпунктов с отдельными промптами, выполняй их последовательно сверху вниз в том же run.
- Если среда сначала переводит этапный запрос в `plan mode`, такой план считается промежуточным артефактом и не подменяет исходный промпт этапа при приемке.
- Если этап не принят, править нужно соответствующий раздел именно в `full-run.md`, после чего запускать новый чистый run.
- Устное сопровождение ниже можно проговаривать почти дословно во время презентации.

## Общий принцип для всех этапов

- Каждый этап внутри run должен описывать только текущее состояние проекта и следующий шаг.
- Первый внутренний этап уже начинает собирать сам проект внутри отдельного автономного workspace.
- Каждый следующий промпт предполагает, что предыдущие этапы в этом же run уже выполнены.
- Начиная с подпункта `1.3`, промпты не повторяют отдельно требование читать `task-brief.md`, `project-concept.md` и проектные `rules`: после подпункта `1.2` это считается уже встроенным рабочим поведением.
- Каждый внутренний этапный промпт должен быть самодостаточным: не отсылай его к другим разделам `full-run.md`, `implementation-plan.md`, `stage-validation.md` или к внешнему контуру как к источнику обязательных действий; не ссылайся внутри самого промпта на номера этапов, подпунктов и другую внешнюю нумерацию. Допустимы только ссылки на артефакты и состояние, уже существующие внутри текущего `run`.
- Начиная с подпункта `1.3`, внутри `run` должны уже существовать `.cursor/agents/checkpoint-validator.md` и `.cursor/commands/checkpoint-validator-after-code.md`; дальше они используются как постоянный контур независимой приемки после кодовых этапов.

## Приемка после кодовых этапов

- Этот раздел относится только к кодовым этапам, выполняемым внутри автономного `run` или `demo-live`.
- Он не предписывает запускать проверяющего агента при редактировании, ревью и отладке самих `md`-документов во внешнем `meta-workspace`.
- Этапы 2-8 сначала могут проходить через `plan mode`, если этого требует среда.
- Это не меняет источник приемки: быстрый check и проверяющий агент сверяют результат с исходным текстом текущего этапного промпта из этого файла.
- `Этап 2` завершает bootstrap внутреннего прогона: для него достаточно прямой проверки запускаемого каркаса, конфигурационных заготовок и команды старта.
- Начиная с `Этапа 3`, сразу после написания или изменения кода в рамках этапа родительский агент в том же workflow сначала запускает быстрый check, затем запускает `.cursor/commands/checkpoint-validator-after-code.md`.
- Эта команда должна передавать свежий контекст в `.cursor/agents/checkpoint-validator.md` и не опираться на историю рассуждений родительского агента.
- Если проверяющий агент выдал замечания, внеси минимальные исправления, снова запусти быстрый check и затем снова запусти команду проверки.
- Повторяй этот цикл, пока проверяющий агент не вернет `PASS` без замечаний.
- Этот приемочный цикл считается частью самого выполнения этапа. Во внешнем ревью не требуется отдельно повторно запускать его или требовать специально сохраненный артефакт `PASS`, если текущее состояние `run` соответствует этапу и нет признаков незакрытых замечаний.
- После `Этапа 2` отдельный `PASS` от проверяющего агента еще не обязателен: важно, что сам приемочный контур уже создан на подпункте `1.3` и готов к следующим кодовым этапам.
- Переход к следующему кодовому этапу начиная с `Этапа 3` допустим только после отдельного `PASS`.

### Подпункт 1.3. Проверяющий агент

Отдельный промпт на этом шаге не нужен. Этот подпункт выполняется вручную внутри `run`.

Что сделать:

- создай `.cursor/agents/checkpoint-validator.md`
- создай `.cursor/commands/checkpoint-validator-after-code.md`
- опиши проверяющего агента как независимый контур приемки в режиме только чтения со свежим контекстом
- зафиксируй источники истины: `task-brief.md`, `project-concept.md`, текущий этапный запрос, измененные файлы и результат быстрой проверки
- зафиксируй, что промежуточный план из режима планирования считается только вспомогательным артефактом и не подменяет текущий этапный запрос
- зафиксируй формат ответа: `вердикт`, `что подтверждено`, `что нарушено`, `риск для основного сценария`, `минимальное исправление`, `можно ли переходить дальше`
- зафиксируй строгий формат ответа с буквальными полями: `вердикт` только со значениями `PASS`, `WARN` или `FAIL`
- зафиксируй в `.cursor/commands/checkpoint-validator-after-code.md`, что сразу после написания или изменения кода на каждом кодовом этапе основной агент сначала запускает быстрый check, затем запускает проверяющего агента, исправляет замечания, повторно запускает быструю проверку и затем снова запускает проверку, пока не получит `PASS` без замечаний

Ограничения:

- не меняй без необходимости уже созданные документы и правила
- проверяющий агент не должен переписывать критерии приемки и расширять трактовку задачи
- используй русские формулировки в описании, кроме точных имен технологий, команд, путей и API

Критерий готовности:

- в рабочем каталоге есть `.cursor/agents/checkpoint-validator.md`
- в рабочем каталоге есть `.cursor/commands/checkpoint-validator-after-code.md`
- проверяющий агент описан как отдельный независимый контур приемки
- в описании агента и команды явно зафиксирован цикл повторных проверок до `PASS` без замечаний

### Общий критерий готовности этапа 1

- в рабочем каталоге есть `task-brief.md` и `project-concept.md`
- в рабочем каталоге есть `.cursor/rules/chess-coach-mvp.mdc`, `.cursor/rules/stockfish-vs-llm.mdc`, `.cursor/agents/checkpoint-validator.md` и `.cursor/commands/checkpoint-validator-after-code.md`
- документы задают понятные границы MVP
- правила помогают агенту не выходить за границы задачи и не смешивать роли движка и `LLM`
- проверяющий агент описан как отдельный независимый контур приемки
- для следующих кодовых этапов подготовлен встроенный цикл проверки через быстрый check и запуск команды `checkpoint-validator-after-code`

### Устное сопровождение

- На первом шаге я не прошу ИИ сразу писать код. Сначала я создаю автономный контекст проекта в документах, чтобы дальнейшие шаги были управляемыми и воспроизводимыми.
- Затем отдельным коротким запросом выношу часть управления агентом в правила.
- Подпункт `1.3` я не отдаю агенту: здесь я вручную добавляю проверяющего агента и команду его запуска, чтобы сохранить независимый контур приемки для следующих кодовых этапов.
- Если захочу отдельно показать `hooks`, это можно сделать позже на кодовом этапе вокруг быстрой проверки, не усложняя стартовый контур.
- Сразу после этого шага удобно точечно показать `skill` `create-rule` и быстро добавить или уточнить одно узкое проектное правило.

### Подпункт 2.3. Подготовка к взаимодействию с `Cursor Cloud`

#### Промпт

```text
В проекте уже есть минимальный запускаемый каркас, конфигурация через `.env` и шаблон `.env.example`.

цель:
- подготовить проект к надежному взаимодействию с облачной `LLM` через `Cursor Cloud`, не смешивая этот шаг с продуктовой логикой объяснений

границы задачи:
- выбери один конкретный способ обращения к `Cursor Cloud` и зафиксируй его в коде и документации
- подготовь отдельный узкий слой для интеграции с облачной моделью, например `cursor_cloud.py`, `llm_client.py` или эквивалентный модуль
- используй такой подход: API key хранится в `.env`, base URL берется из конфигурации, явная модель задается опционально, а при пустом значении проект не хардкодит фиктивный `model`
- если нет сильной причины сделать иначе, используй совместимую схему переменных `CURSOR_CLOUD_AGENTS_API_KEY`, `CURSOR_CLOUD_BASE_URL`, `CURSOR_CLOUD_MODEL` и `CURSOR_CLOUD_TIMEOUT_SEC`
- если `CURSOR_CLOUD_MODEL` пустой, `None` или равен `default`, подготовь код так, чтобы на этапе реального вызова поле `model` можно было не передавать вовсе, оставляя выбор модели на стороне `Cursor Cloud`
- если на предыдущем шаге уже появились общие заглушки вроде `LLM_API_KEY`, `LLM_BASE_URL`, `LLM_MODEL`, `LLM_VENDOR_*` или UI-метрики для них, не оставляй их основным источником истины для этого сценария: либо замени их на `CURSOR_CLOUD_*`, либо сделай явное и единое отображение `CURSOR_CLOUD_*` в внутренние настройки без дублирующихся веток конфигурации
- приведи к одной схеме не только модуль настроек, но и все места, где проект показывает статус конфигурации: `Streamlit`-экран, health-check, диагностические сообщения, `README.md` и `.env.example` должны смотреть на один и тот же набор реальных переменных
- добавь отдельную легкую проверку готовности подключения, чтобы до этапа интеграции объяснений можно было убедиться, что конфигурация читается корректно и проект готов к сетевому вызову
- обнови `.env.example` и `README.md`, чтобы было понятно, какие переменные нужно заполнить для `Cursor Cloud` и как проверить готовность интеграции
```
