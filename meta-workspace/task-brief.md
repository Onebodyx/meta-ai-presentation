# Постановка задачи

## Контекст
Нужно подготовить и в рамках одной живой демонстрации показать руководителю небольшой проект, собранный с нуля с помощью ИИ-агента в `Cursor`.

Цель демонстрации двойная:

- показать, что ИИ-агенты многократно ускоряют разработку;
- обязательно показать, что реальная эффективность достигается только при наличии инженерного мышления, умения ставить задачи, декомпозировать работу, управлять контекстом и использовать дополнительные инструменты вроде `rules`, `hooks`, `skills` и `MCP`.

У руководителя есть интерес к шахматам, поэтому тема проекта должна быть связана с шахматами и при этом быть наглядной в рамках презентации длительностью не более одного часа.

## Формат рабочих контуров
Проект организуется в двух слоях с разной ролью:

- корень текущего проекта используется как внешний контур подготовки;
- во внешнем контуре создаются и корректируются master-документы, `full-run.md` как единый сценарий промптов, критерии проверки и заметки ревью;
- для live-demo используется отдельный автономный workspace `demo-live`, который открывается в `Cursor` как единственный рабочий корень;
- для репетиций и проверки используются отдельные изолированные run-ы вне дерева внешнего проекта, например `demo-runs/run-003`;
- каждый run моделирует одну полную презентацию и может быть остановлен после нужного этапа для проверки;
- внутренний контур не должен знать о существовании внешнего: в его документах, правилах и коде не должно быть ссылок на подготовительный слой;
- каждый новый прогон должен стартовать как отдельный автономный workspace без проектных артефактов; технические служебные файлы вроде `.git` допустимы, если они не подменяют документы, правила, команды и код проекта;
- все документы, `rules`, `agents`, `commands`, `hooks`, `skills` и код приложения должны появляться внутри каждого внутреннего прогона только через промпты, подготовленные во внешнем контуре.
- во внешнем `meta-workspace` master-документы, сценарии и критерии проверяются напрямую; независимый проверяющий субагент обязателен только внутри автономного `run` или `demo-live` после завершенного кодового этапа.

## Поэтапная проверка прогонов
Внешний контур должен уметь проверять не только полный результат, но и каждый этап отдельно.

Для этого:

- для каждого этапа есть отдельный раздел в `full-run.md`;
- для каждого этапа фиксируется свой критерий готовности и набор ожидаемых артефактов;
- внешний контур соотносит состояние конкретного run-а с промптами из `full-run.md` до нужного этапа включительно;
- если этап не принят, корректируется соответствующий раздел в `full-run.md`, после чего запускается новый чистый run.

## Что необходимо сделать
Нужно реализовать демонстрационный MVP сервиса для обучения игре в шахматы.

Минимальный обязательный сценарий:

- пользователь делает ходы в интерфейсе;
- противник делает ответные ходы с помощью шахматного движка;
- сервис показывает несколько сильных кандидатных ходов для игрока;
- сервис подробно объясняет плюсы, минусы, планы и риски для этих ходов;
- объяснения даются в понятной обучающей форме, а не только в виде сухой оценки позиции.

## Принципиальное распределение ответственности
Система должна быть построена на четком разделении ролей:

- `Stockfish` или другой шахматный движок отвечает за легальность ходов, силу игры и оценку позиции;
- облачная `LLM` отвечает за объяснение: почему ход хороший, какие есть идеи, какие риски возникают и как человеку мыслить в позиции;
- агент в `Cursor` отвечает за быстрое построение проекта, написание кода, интеграцию компонентов, тестовые проверки и UI под управлением человека.

`LLM` не должна быть источником шахматной истины. Она не принимает решения о легальности ходов и не подменяет собой движок.

## Ограничения
- На всю демонстрацию есть не более одного часа.
- Проект должен собираться быстро и быть достаточно надежным для live-demo.
- Допустимы локальные установки и локальный запуск сервера/движка.
- Для объяснений доступен API облачной модели.
- Для первой версии выбирается максимально простой и надежный стек: `Streamlit + Python + Stockfish + cloud LLM API`.

## Главный тезис презентации
Демонстрация должна доказывать не только то, что ИИ ускоряет разработку, но и то, что результат напрямую зависит от качества инженерного управления агентом.

Во время показа нужно явно подсветить:

- как формулируется задача для агента;
- как большая задача декомпозируется на узкие этапы;
- почему важно ограничивать контекст и держать единый источник правды;
- почему надежная архитектура строится на разделении ответственности между компонентами;
- почему дополнительные инструменты `Cursor` повышают качество и предсказуемость результата.

## Что обязательно продемонстрировать в `Cursor`
Во время живого показа недостаточно просто упомянуть дополнительные инструменты. Нужно явно показать, в какой момент и зачем используются:

- старт из отдельного автономного workspace, внутри которого проектные артефакты появляются пошаговыми промптами из `full-run.md`;
- возможность остановиться после любого этапа и проверить промежуточный результат по заранее заданному критерию;
- `rules` как постоянные проектные ограничения и защита от расползания задачи;
- `hooks` как автоматические проверки после изменений;
- независимый проверяющий субагент, который начиная со следующего кодового этапа после базового runnable-каркаса запускается внутри workflow родительского агента в автономном `run` или `demo-live` сразу после написания или изменения кода, отдельно сверяет результат с проектными документами и с полным текстом текущего этапного промпта из `full-run.md`, который ему передает родительский агент, а затем вызывается повторно до тех пор, пока не останется замечаний и не будет получен `PASS`;
- `skills` как способ упаковать повторяемые экспертные действия в переиспользуемый рабочий процесс;
- `MCP` как способ получать актуальную документацию и снижать риск ошибок из-за устаревших знаний модели.

## Критерии успеха
Результат можно считать успешным, если к концу работы есть локально запускаемый MVP, который:

- собран внутри автономного workspace, открываемого отдельно от внешнего контура;
- может быть воспроизведен и проверен не только целиком, но и после каждого этапа на отдельном чистом прогоне;
- после bootstrap-этапа 2 подтверждается прямым запуском каркаса, а начиная со следующего кодового этапа внутри автономного `run` или `demo-live` сразу после написания или изменения кода проходит быстрый автоматический check и встроенную независимую проверку свежим проверяющим субагентом с повторением цикла до финального `PASS` без замечаний; для внешнего ревью отдельный повторный запуск или специально сохраненный артефакт `PASS` не требуются, если итоговое состояние run-а соответствует этапу;
- позволяет сыграть короткий фрагмент партии против движка;
- не допускает нелегальных ходов;
- может показать несколько сильных продолжений для текущей позиции;
- сопровождает варианты понятным текстовым объяснением от `LLM`;
- содержит отдельные и понятные моменты демонстрации `rules`, `hooks`, `skills` и `MCP`;
- дает материал для обсуждения инженерных практик работы с ИИ-агентами.

## Что не входит в первую версию
В первую версию сознательно не включаются:

- регистрация пользователей;
- база данных и сохранение истории;
- полноценный продакшен-деплой;
- сложная многопользовательская логика;
- попытка сделать `LLM` самостоятельным шахматным противником.

## Рабочий ориентир для дальнейших агентов
Все последующие задачи, решения и изменения должны проверяться на соответствие этой постановке. Если появляется соблазн расширить scope, приоритет остается за надежным, узким и хорошо демонстрируемым MVP.

Такая проверка кодовых этапов внутри автономного `run` или `demo-live`, начиная со следующего шага после bootstrap-этапа 2, должна опираться не только на контекст родительского агента, но и на свежий запуск независимого проверяющего субагента сразу после написания или изменения кода. Этот субагент читает текущие артефакты проекта, сверяет их с полным текстом текущего этапного промпта из `full-run.md`, который ему передает родительский агент, выносит отдельный вердикт и затем вызывается повторно до тех пор, пока не будет получен `PASS` без замечаний.

Во внешнем `meta-workspace` этот запуск не требуется для проверки самих master-документов, сценариев и критериев; там также не требуется отдельный повторный запуск или специально сохраненный артефакт `PASS`, если итоговое состояние run-а соответствует этапу и не показывает признаков незакрытых замечаний.
